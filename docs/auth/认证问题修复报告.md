## 🐛 问题 1: 旧的 Magic Link 仍然可以登录

### 问题描述
用户点击邮件中的 Magic Link 登录成功后，再点击同一邮箱收到的**以前的邮件链接**，仍然可以成功登录。

### 安全风险
- ⚠️ **高风险**: Magic Link 应该是一次性的
- ⚠️ 旧链接可能被截获或泄露
- ⚠️ 违反了 Magic Link 的安全设计原则

### 根本原因

**文件**: `app/[locale]/auth/callback/route.ts`

**问题代码**:
```typescript
// ❌ 错误的逻辑顺序
const supabase = await createClient();

// 1. 首先检查是否已登录
const { data: { user: existingUser } } = await supabase.auth.getUser();
if (existingUser) {
  console.log('[auth/callback] already logged in, redirecting');
  return buildRedirect(request, `${origin}${next}`);
  // ❌ 直接重定向，没有验证 token！
}

// 2. 然后才验证 token
if (tokenHash && type) {
  const { data, error } = await supabase.auth.verifyOtp({
    token_hash: tokenHash,
    type: type as 'magiclink' | 'email',
  });
  // ...
}
```

**问题流程**:
```
1. 用户点击新的 Magic Link
   ↓
2. verifyOtp() 成功，创建 session
   ↓
3. 用户已登录
   ↓
4. 用户点击旧的 Magic Link
   ↓
5. 检测到已登录 ✅
   ↓
6. 直接重定向到 dashboard ❌
   ↓
7. 没有验证旧 token 是否有效 ❌
```

### 修复方案

**核心原则**: **始终验证 token，即使用户已登录**

**修复后的代码**:
```typescript
// ✅ 正确的逻辑顺序
const supabase = await createClient();

// 1. 如果有 token，必须先验证
if (tokenHash && type) {
  console.log('[auth/callback] verifying OTP with token_hash');
  const { data, error } = await supabase.auth.verifyOtp({
    token_hash: tokenHash,
    type: type as 'magiclink' | 'email',
  });
  
  if (error) {
    console.error('[auth/callback] verifyOtp error:', error.message);
    
    // ✅ 如果 token 无效但用户已登录，先登出
    const { data: { user: existingUser } } = await supabase.auth.getUser();
    if (existingUser) {
      console.log('[auth/callback] logging out user due to invalid token');
      await supabase.auth.signOut();
    }
    
    return buildRedirect(request, `${origin}/${locale}/login?message=link_expired`);
  }
  
  if (data?.user) {
    console.log('[auth/callback] verifyOtp success, user:', data.user.email);
    return buildRedirect(request, `${origin}${next}`);
  }
}

// 2. 如果没有认证参数，才检查是否已登录
const { data: { user: existingUser } } = await supabase.auth.getUser();
if (existingUser) {
  console.log('[auth/callback] no auth params but already logged in');
  return buildRedirect(request, `${origin}${next}`);
}
```

**修复后的流程**:
```
1. 用户点击新的 Magic Link
   ↓
2. verifyOtp() 成功 ✅
   ↓
3. 创建 session，用户已登录
   ↓
4. 用户点击旧的 Magic Link
   ↓
5. 尝试 verifyOtp() 旧 token
   ↓
6. Supabase 返回错误: "Email link is invalid or has expired" ✅
   ↓
7. 检测到用户已登录
   ↓
8. 执行 signOut() 登出用户 ✅
   ↓
9. 重定向到登录页，显示 "链接已过期" ✅
```

### 安全改进

- ✅ **Token 验证优先**: 始终先验证 token
- ✅ **一次性使用**: 旧 token 会被 Supabase 拒绝
- ✅ **强制登出**: 使用无效 token 时会登出当前用户
- ✅ **明确提示**: 显示 "链接已过期" 错误信息

## 🐛 问题 2: 链接过期错误信息不会清除

### 问题描述
在显示 "链接已过期" 的登录页面上，用户重新发送 Magic Link 后，错误信息仍然显示在页面上。

### 用户体验问题
- ❌ 用户看到错误信息，可能认为新邮件也失败了
- ❌ 造成困惑和不信任
- ❌ 需要手动刷新页面才能清除

### 根本原因

**问题 1**: URL 参数持久化
```
用户访问: /en/login?message=link_expired
         ↓
页面显示: "链接已过期"
         ↓
用户点击 "发送 Magic Link"
         ↓
URL 仍然是: /en/login?message=link_expired ❌
         ↓
错误信息仍然显示 ❌
```

**问题 2**: 服务端渲染的错误信息
```typescript
// app/[locale]/login/page.tsx
const { message } = await searchParams;
const messageText = message === 'link_expired' ? t('linkExpired') : null;

// ❌ 这是服务端渲染的，客户端操作不会改变它
{messageText && (
  <div className="mb-6 border-l-2 border-destructive pl-4">
    <p className="text-sm text-destructive">{messageText}</p>
  </div>
)}
```

### 修复方案

**方案 1**: 在表单提交时清除 URL 参数

**文件**: `components/auth/login-form.tsx`

```typescript
// ✅ 已有的清除函数
function clearUrlMessage() {
  if (typeof window === 'undefined') return;
  const url = new URL(window.location.href);
  if (url.searchParams.has('message')) {
    url.searchParams.delete('message');
    window.history.replaceState({}, '', url.toString());
  }
}

// ✅ 在所有表单提交时调用
const handleMagicLink = async (e: React.FormEvent) => {
  e.preventDefault();
  setLoading(true);
  setMessage(null);
  clearUrlMessage(); // ✅ 清除 URL 参数
  // ...
};

const handlePasswordLogin = async (e: React.FormEvent) => {
  e.preventDefault();
  setLoading(true);
  setMessage(null);
  clearUrlMessage(); // ✅ 清除 URL 参数
  // ...
};
```

**方案 2**: 在切换登录模式时清除

```typescript
// ✅ 切换到密码登录
<button
  onClick={() => { 
    setMode('password'); 
    setMessage(null);
    clearUrlMessage(); // ✅ 清除 URL 参数
  }}
>
  {t('passwordLogin')}
</button>

// ✅ 切换到 Magic Link
<button
  onClick={() => { 
    setMode('magic-link'); 
    setMessage(null);
    clearUrlMessage(); // ✅ 清除 URL 参数
  }}
>
  {t('magicLink')}
</button>
```

### 修复后的流程

```
1. 用户点击过期的 Magic Link
   ↓
2. 重定向到: /en/login?message=link_expired
   ↓
3. 页面显示: "链接已过期" ⚠️
   ↓
4. 用户输入邮箱，点击 "发送 Magic Link"
   ↓
5. handleMagicLink() 执行
   ↓
6. clearUrlMessage() 清除 URL 参数 ✅
   ↓
7. URL 变为: /en/login (无参数) ✅
   ↓
8. 服务端错误信息不再显示 ✅
   ↓
9. 显示客户端成功消息: "请查收邮件" ✅
```

### 用户体验改进

- ✅ **自动清除**: 提交表单时自动清除错误
- ✅ **即时反馈**: 显示新的成功消息
- ✅ **无需刷新**: 不需要手动刷新页面
- ✅ **清晰状态**: 用户知道新邮件已发送
